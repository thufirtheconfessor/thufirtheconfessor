Thufir the Confessor

Introduction

Thufir is a barcode reader to receive files through a series of 2D barcodes. The programs to send files are easy to port and even short enough to be typed in.

Thufir's intended use is to copy files when access to a computer is limited to a keyboard and display but accessing files and running programs are possible. Currently there are senders for HTML5 browsers, Java, and Excel.

The steps to copy a file with Thufir are

  • identify which senders can be run on the computer
  • type in or copy to the computer a sender
  • set the sender and receiver parameters
  • point the phone's camera to the computer's screen
  • start the sender
  • hold while the file is being received
  • the file should be in the folder <sdcard>/thufir/.

It's easier to try out Thufir first on a computer without restrictions.

Senders

While all the senders are equally capable and the HTML5 one is the easiest to use, the most important factor is which one can be run on the computer.

After the sender has been chosen it must be typed in or copied to the computer. The sender can be typed in directly but it's error prone, so in the vein of type-in programs of old computer magazines a shorter program is provided to check the correctness of other programs. With this the CRC32 hash of every line of the typed-in program is computed and can be compared to the provided correct values. As the hashes are chained from the first line to the last, matching hashes indicate likely correctness of the line of the hash and all previous lines. example

With this chaining

  • if the last hash of the typed in program matches the correct one, the program is likely correct.
  • if there's at least one typo, all the hashes before the first typo match, and the hashes of the first typo and all subsequent lines likely mismatch. example
  • missing lines don't affect the hashes of preceding lines, so partial programs can be checked after every line typed in.

While the crc program can be checked by itself and all matching hashes mean likely correctness, a crc program with errors in it may produce no or any output, so incrementality likely doesn't hold on the crc program itself. example

Settings

The senders all have common parameters but usually only the from and the file can be set easily at every execution, the rest must be set in the source code, and must be recompiled if that's applicable. As the programs are easily made incorrect by small mistakes, it's advised to make backup copies of known correct programs before modifying them.

  • from: file offset to start the transmission from. Thufir can continue the reception of interrupted transmissions and this specifies how much has been already received. For new files specify 0, for partially received files use the value shown on the progress bar.
  • file: file to send.
  • br and da: bright and dark colors of the barcodes. Some displays and cameras may have better response to an inverted color scheme, or more specific colors.
  • tw and th: width and height of the individual outer tracking cells. Larger tracking cells are easier to find on an image and provide more precision.
  • cw and ch: width and height of the individual inner data cells.
  • dw and dh: width and height of all the data cells, expressed as 16x16 data blocks.
  • de: delay in ms between barcodes shown.
  • id: delay in ms to wait after the sender started to show the first barcode, to allow time to move the phone into position.
  • xx and yy: java only parameters to control the position of the barcodes on the screen.

Every barcode contains 28*dw*dh-1 bytes of data, so the speed of the transmission is 1000*(28*dw*dh-1)/de bytes per second.

HTML5

These are most likely the easiest senders to use as they only require a text editor and a compatible browser, which are commonly available.

There are the crc.html and 2 senders

  • canvas.html exposes only parameters file and from on the user interface
  • canvas-ui.html has all the parameters on the interface for easy experimentation.

Java

To run the java sender a java compiler and runtime environment must be available. Supported versions are standard edition 1.2 and up. To check their versions run java -version and javac -version.

There are the Crc and 3 senders

  • AnsiStdOut: for character terminals with ANSI escape code support
  • Awt: for graphical terminals
  • StdOut: for character terminals without ANSI escape code support

To compile one of the programs run javac [file.java].

To check crc on a file run java Crc [file.java].

To send a file run java [AnsiStdOut|Awt|StdOut] [from] [file].

Excel

The Excel sender is the cruftiest of them. To run this an Excel with macros enabled is needed. There's only one source file in this sender and it contains the crc as well. If all the lines before the comment Rem sender has been typed in, running the sub crc will copy the hashes of all the lines from Module1 to the first sheet.

After all the lines are typed in

  • go to the second sheet
  • use the cell A1 as the parameter from
  • use A2 to hold the path to the file
  • create two buttons in the first line
  • assign one button to the sub sf to select files
  • assign the other button to the sub st to start sending
  • press a lot of escapes to stop it before it runs down

Others

Other sources can be shown on Thufir's internal crc checker. Files put into the folder <sdcard>/thufir/sources/ show up in the type-in menu.

Phone settings

The settings on the receiving side are grouped into two. Camera settings are provided by the phone, and they vary. Program settings affect how images are taken and processed, and the look of the program.

Settings in red are available, but not recommended.

The more important camera settings are

  • Camera: camera to use, mainly back and front
  • Picture size: resolution of pictures taken
  • Preview FPS: frame rate of the preview
  • Preview size: resolution of the preview.

Program settings are

  • Blob size: method to measure blobs. area is more precise, perimeter is faster
  • Brightness scheme: barcode colors to look for: bright on dark looks for bright barcodes on a dark background, dark on bright for dark barcodes on a bright background, auto for both, which is slower
  • Buttons' position: corner to put buttons in
  • Diagnostics: turns on-screen diagnostics on and off
  • Distortion correction: Larger barcodes with small data cells may be read more reliably with distortion correction, but it can be quite slow, so it may not help with overall transmission rate. no only corrects perspective. some tries to correct some lens distortion. some-s is single threaded, while some-ml, some-mm, and some-mlm are multithreaded with increasing parallelisation.
  • Filter: whole-image filter to apply before any image processing. These may help on character terminals. Max filters replace every pixel with the brightest pixel in its neighborhood, to be used with bright on dark barcodes. Similarly min filters choose the darkest pixels, and are intended for dark on bright barcodes. The dimensions specify the size of the neighborhood
  • Image source: picture takes pictures, which has better quality, preview uses the viewfinder preview, which has a higher frame rate
  • Parallelism: number of threads to process the images
  • Threshold: brightness to use in partitioning the image. Brightness values less than threshold are classified as dark, all else as bright
  • Threshold mode: manual mode partitions the images by value of the setting threshold, which is faster, and auto mode ignores the setting threshold, and a threshold is calculated for every image based on its brightness histogram.

Diagnostics

The on-screen diagnostics show some raw measurements to help find the correct settings.

Running the diagnostics have a similar effect to receiving files, the screen is forced on, and pictures are taken, and processed.

Some measurements are performed below and around the cross-hair on the image overlay, and with diagnostics turned on, the cross-hair is movable by touch.

The measurements are:

  • min. frame, avg. frame, max. frame: aggregates of times between two pictures taken by the camera
  • dropped frames: ratio of images dropped per images take
  • min. proc., avg. proc., max. proc.: aggregates of image processing times
  • threshold: running average of the thresholds used in the partitioning
  • brightness: brightness of the pixel under the cross-hair, and its classification
  • contrast: contrast around the cross-hair.

Protocol

Thufir transmits a single file through a series of barcodes. Every barcode contains data and CRC32 hashes. The data is concatenated to form the file, while the hashes are used to check its integrity. The hashes are chained to order the barcodes.

The data stream is the concatenation of

  • the length of the name of the file as 1 byte
  • every ASCII code of the file name as a byte
  • the size of the file as 4 bytes
  • the content of the file
  • padding to make the length of the stream divisible by the data size a the barcodes.

Every barcode have two parts, the outer tracking, and the inner data cells. The tracking cells identify the barcode in the image, determine the amount of data it contains, and locate the data cells. Data cells are further differentiated into 3 types:

  • clock cells at the corners and the middle of the sides ensure different consecutive barcodes
  • hash cells for integrity
  • actual data cells of the stream.

The parameters dw and dh specify how wide and tall barcodes should be, set in 16x16 blocks of data cells. For every 28 bytes of clock and data there are 4 bytes of crc, which added together is a complete data block. And there are 8 clock cells in a barcode. So a barcode contains 28*dw*dh-1 bytes of data.

Anatomy of a barcode

  • black: tracking
  • silver: width and height
  • blue: data
  • red: clock
  • lime: hash

Speed up

With some fiddling Thufir can reach not completely useless speeds.

  • High brightness and contrast of the display separate black and white more and may also make transitions faster.
  • Manual threshold mode and brightness scheme process images faster.
  • Higher parallelism can speed up image processing, but too high values may even slow it down.
  • The delay between barcodes should be at least twice as much as the times between captured frames, sometimes much higher, especially in preview mode.
  • Larger tracking cells provide more precision.
  • Distortions can be decreased by centering and aligning the barcode in the image.
  • Holding the camera too close to the display increases the distortions.
  • Anything longer than tiny files require some contraption to hold the phone in place.
